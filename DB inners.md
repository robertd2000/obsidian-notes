Если углубиться, то в PostgreSQL всё вращается вокруг одной фундаментальной абстракции — **страницы (page)**. Это атомарная единица хранения и I/O, обычно 8 KB. Ни таблицы, ни строки напрямую «с диска» не читаются — только страницы.

### Как устроена страница

Страница — это не просто кусок байтов. У неё есть чёткая внутренняя структура:

- в начале — заголовок страницы (LSN, флаги, указатели на свободное место),
    
- дальше — массив указателей (line pointers),
    
- в конце — сами строки (tuples), которые растут «с хвоста».
    

Это сделано так, чтобы можно было перемещать строки внутри страницы без изменения ссылок на них. В указателе хранится смещение внутри страницы, а не сама строка.

Строка (heap tuple) внутри страницы — это тоже не просто payload. У неё есть:

- xmin / xmax (MVCC-метаданные),
    
- битовые маски NULL-полей,
    
- сами значения колонок.
    

Из-за MVCC UPDATE не перезаписывает строку. Вместо этого:

- старая версия остаётся,
    
- новая версия записывается (часто в ту же страницу, если есть место),
    
- старая помечается через xmax.
    

Если места нет — новая версия уезжает в другую страницу, и появляется chain (HOT / non-HOT updates).

### Как таблица хранится на диске

Таблица — это просто файл (или несколько файлов, если она большая). Этот файл — это последовательность страниц:

```
[Page 0][Page 1][Page 2]...[Page N]
```

Никакой «структуры таблицы» внутри файла нет — это heap (куча). Поэтому:

- строки не отсортированы,
    
- вставки идут в первое подходящее свободное место (через free space map).
    

Чтобы не сканировать весь файл в поиске свободного места, PostgreSQL использует FSM (Free Space Map) — отдельную структуру, которая говорит, в каких страницах есть место.

### Как происходит INSERT

Когда ты делаешь INSERT:

1. Система ищет страницу с достаточным свободным местом (через FSM).
    
2. Загружает страницу в shared buffers (если её там нет).
    
3. Добавляет tuple в конец страницы.
    
4. Добавляет line pointer.
    
5. Генерирует WAL-запись (что именно изменилось в странице).
    
6. WAL пишется на диск (fsync при commit).
    
7. Страница остаётся в памяти и позже будет записана на диск.
    

Ключевой момент: **страница не обязана сразу попасть на диск** — WAL уже гарантирует восстановление.

### Как работает индекс (B-Tree)

Индекс — это отдельный файл, тоже разбитый на страницы, но структура уже не heap, а дерево.

У B-Tree есть:

- root page,
    
- внутренние страницы (с ключами и указателями),
    
- leaf pages (с ключами и ссылками на heap).
    

В leaf-странице хранится:

- ключ (например, значение колонки),
    
- TID (tuple identifier) → (page_id, offset внутри страницы).
    

#### Вставка в индекс

При INSERT строки:

1. Из tuple извлекается индексируемое значение.
    
2. Идёт поиск нужного leaf-узла:
    
    - начиная с root,
        
    - спускаясь по страницам (binary search внутри страницы).
        
3. В leaf вставляется (key, TID).
    
4. Если страница переполнена — происходит split:
    
    - страница делится,
        
    - средний ключ поднимается вверх,
        
    - возможно каскадное расщепление до root.
        

Это классическое поведение B-Tree.

### Как происходит SELECT без индекса

Если нет индекса — делается Seq Scan:

1. PostgreSQL читает страницы таблицы по порядку.
    
2. Для каждой страницы:
    
    - проходит по line pointers,
        
    - проверяет видимость строки (MVCC: xmin/xmax),
        
    - применяет WHERE.
        

Это максимально простой, но дорогой путь.

### Как происходит SELECT с индексом

Если есть индекс:

1. Планировщик выбирает Index Scan.
    
2. По индексу ищется нужный ключ:
    
    - root → internal → leaf.
        
3. В leaf находятся TID’ы.
    
4. Для каждого TID:
    
    - читается соответствующая страница heap,
        
    - извлекается строка,
        
    - проверяется видимость (MVCC всё равно нужен!).
        

Важно: индекс **не хранит сами строки**, только ссылки. Поэтому почти всегда есть дополнительный random I/O в heap.

### Почему это может быть медленно

Если у тебя много попаданий по индексу:

- ты прыгаешь по heap страницам (random access),
    
- это хуже, чем последовательное чтение.
    

Поэтому PostgreSQL иногда выбирает Seq Scan даже при наличии индекса.

### Index Only Scan

Оптимизация: если все нужные поля есть в индексе + visibility map говорит, что страница «чистая», то можно не ходить в heap.

Visibility Map — это ещё одна структура:

- отмечает страницы, где все строки «видимы» (no dead tuples).
    

Тогда:

- читается только индекс,
    
- heap не трогается.
    

### Как идёт запись (в общем виде)

Любая операция (INSERT/UPDATE/DELETE):

1. Меняет страницу в памяти (shared buffers).
    
2. Генерирует WAL-запись (лог изменений).
    
3. WAL пишется на диск.
    
4. Позже background процессы:
    
    - записывают грязные страницы на диск,
        
    - делают checkpoint.
        

Это ключевой принцип:  
**WAL → потом данные**, а не наоборот.

### Итоговая картина

Если собрать всё вместе:

- таблица = heap из страниц без порядка,
    
- строка = версия с MVCC-метаданными,
    
- индекс = отдельное B-Tree с ссылками на строки,
    
- INSERT = запись в страницу + индекс + WAL,
    
- SELECT:
    
    - без индекса → линейный проход по страницам,
        
    - с индексом → дерево → ссылки → heap,
        
- durability обеспечивается через WAL, а не немедленную запись страниц.
    

Если хочешь, могу дальше углубиться в конкретные вещи — например:

- HOT updates и почему они уменьшают нагрузку на индексы
    
- как именно устроен WAL (record types, LSN, page redo)
    
- или как planner выбирает между Seq Scan и Index Scan

Если отбросить SQL-абстракции, то в PostgreSQL таблица — это **heap-файл**, то есть просто неупорядоченный набор страниц. Никакой «табличной структуры» на уровне файла нет — только последовательность страниц фиксированного размера.

### Как физически лежит таблица

У каждой таблицы есть свой файл (или несколько, если она большая — PostgreSQL режет их по ~1 GB сегментами). Внутри файла:

```
[Page 0][Page 1][Page 2] ... [Page N]
```

Каждая страница — это независимый контейнер строк. Между страницами нет связей, нет сортировки, нет индексов — это именно heap.

Когда ты создаёшь таблицу, PostgreSQL просто создаёт пустой файл. Когда вставляешь данные — страницы начинают заполняться.

---

### Что происходит внутри страницы (чуть глубже)

Внутри страницы есть три ключевые зоны:

1. **Page header** — служебные данные (включая LSN для WAL)
    
2. **Line pointer array** — массив «указателей» на строки
    
3. **Tuple data** — сами строки, лежат с конца страницы
    

Важно: line pointer — это стабильная сущность. Даже если строка перемещается внутри страницы, pointer остаётся, меняется только offset.

---

### Как адресуется строка (TID)

Каждая строка в таблице имеет физический адрес:

```
TID = (block_number, offset_number)
```

- block_number → номер страницы (Page N)
    
- offset_number → индекс в массиве line pointers
    

Это и есть **единственный реальный способ найти строку на диске**.

---

### Что происходит при INSERT

Когда ты вставляешь строку:

- PostgreSQL ищет страницу со свободным местом (через FSM)
    
- вставляет tuple в конец страницы
    
- создаёт line pointer
    
- формирует TID
    

Например:

```
Page 42
  offset 3 → tuple (user_id=10, name='Bob')
```

TID = (42, 3)

---

### Как хранятся индексы и как они ссылаются на таблицу

Индекс — это отдельный файл с собственной структурой (например, B-Tree). Он **вообще не содержит строки таблицы**.

В индексе хранится:

```
(key, TID)
```

Например:

```
("Bob", (42, 3))
("Alice", (10, 5))
```

То есть индекс — это просто отображение:

```
значение → где лежит строка
```

---

### Как происходит поиск через индекс

Когда ты делаешь:

```sql
SELECT * FROM users WHERE name = 'Bob';
```

происходит следующее:

1. PostgreSQL идёт в B-Tree индекс:
    
    - root → internal → leaf
        
2. В leaf находит:
    
    ```
    ("Bob", (42, 3))
    ```
    
3. Берёт TID = (42, 3)
    
4. Загружает Page 42 из heap
    
5. По offset 3 достаёт строку
    

То есть индекс **никогда не ведёт напрямую к данным**, он ведёт к адресу в heap.

---

### Важный нюанс: почему это не просто «указатель»

TID — это не pointer в памяти, а **логический адрес внутри файла**. Это важно, потому что:

- данные могут перемещаться между страницами (например, при UPDATE)
    
- старые версии строк остаются (MVCC)
    

---

### Что происходит при UPDATE

UPDATE — это не изменение строки:

1. Старая строка остаётся на месте
    
2. Новая версия записывается (возможно в другую страницу)
    
3. Индексы начинают указывать на новую версию
    

Например:

```
старый TID: (42, 3)
новый TID: (105, 1)
```

Теперь индекс обновляется:

```
("Bob", (105, 1))
```

Старая строка остаётся до VACUUM.

---

### Почему это сложно: MVCC и видимость

Даже если индекс дал тебе TID, PostgreSQL всё равно обязан:

- загрузить строку
    
- проверить её видимость (xmin/xmax)
    

Потому что:

- одна и та же строка может иметь несколько версий
    
- не все версии видимы в текущей транзакции
    

---

### HOT updates (оптимизация)

Иногда PostgreSQL может **не трогать индекс при UPDATE**:

- если изменяются только неиндексируемые поля
    
- новая версия помещается в ту же страницу
    

Тогда создаётся цепочка внутри страницы:

```
(42,3) → (42,7)
```

И индекс продолжает указывать на старый TID, а PostgreSQL уже внутри страницы находит актуальную версию.

Это сильно снижает нагрузку на индексы.

---

### Итоговая картина

Если свести к сути:

- таблица = просто heap-файл из страниц
    
- строка = tuple внутри страницы
    
- адрес строки = TID (page, offset)
    
- индекс = отдельное дерево
    
- индекс хранит только (key → TID)
    
- поиск = индекс → TID → heap → строка
    

И это объясняет почти всё поведение PostgreSQL:

- почему UPDATE дорогой (новая строка + индексы)
    
- почему random I/O при индексах
    
- почему VACUUM нужен
    
- почему иногда Seq Scan быстрее
    

---

Если хочешь, дальше можно разобрать прям «железно»:

- как устроен tuple header (битовые флаги, null bitmap)
    
- как работает visibility map / hint bits
    
- или как именно происходит page split в B-Tree и что пишется в WAL


Да, ты правильно чувствуешь общую идею — но там есть важная деталь: это **не просто массив «один за другим»**, а более хитрая структура, чтобы избежать постоянных копирований и сдвигов.

В PostgreSQL страница устроена как **две растущие навстречу области**.

Представь 8 KB страницы как кусок памяти:

```text
| PageHeader | LP[1] LP[2] LP[3] ...        free space        ... tuple3 tuple2 tuple1 |
```

- слева направо растёт массив **line pointers (LP)**
    
- справа налево кладутся сами **tuple (строки)**
    
- между ними — свободное место
    

---

### Как именно кладутся данные

Когда ты делаешь INSERT:

1. tuple (строка) **сериализуется в бинарный формат**
    
2. кладётся **в конец страницы** (справа)
    
3. создаётся новый **line pointer** (слева)
    

И вот ключевая мысль:

👉 **TID указывает не на сам tuple, а на line pointer**

То есть:

```text
TID = (page=42, offset=3)
```

offset=3 → это **третий элемент в массиве line pointers**, а уже он содержит:

- смещение tuple внутри страницы
    
- длину
    
- статус (live/dead)
    

---

### Почему так сделано

Если бы TID указывал прямо на смещение tuple:

- при любом перемещении строки пришлось бы обновлять ВСЕ индексы
    
- это было бы катастрофически дорого
    

А здесь:

- tuple можно двигать внутри страницы
    
- меняется только offset внутри LP
    
- TID остаётся стабильным
    

---

### Как выглядит line pointer

Упрощённо:

```text
LP[3] → { offset: 7900, length: 120, flags }
```

А tuple лежит где-то в хвосте страницы:

```text
byte 7900 → [tuple data...]
```

---

### Что происходит при DELETE / UPDATE

#### DELETE

- tuple **не удаляется физически**
    
- в LP ставится флаг «dead» (через xmax)
    
- место остаётся занятым до VACUUM
    

---

#### UPDATE

Это интереснее.

1. создаётся **новый tuple** (в той же или другой странице)
    
2. старый tuple помечается как устаревший
    
3. индекс начинает ссылаться на новый TID
    

НО если это HOT update:

- новый tuple кладётся в ту же страницу
    
- старый LP остаётся
    
- создаётся цепочка внутри страницы
    

---

### Что происходит при нехватке места

Когда внутри страницы становится мало места:

- PostgreSQL может делать **compaction**
    
    - сдвигает tuple’ы плотнее к краю
        
    - обновляет offsets в LP
        
- это **не ломает TID**, потому что LP остаётся тем же
    

---

### Это не массив в классическом смысле

Важно понимать разницу:

- ❌ не массив структур `[tuple1, tuple2, tuple3]`
    
- ❌ не linked list
    
- ✅ это:
    
    - массив указателей (LP)
        
    - - отдельная область данных
            

Это ближе к **slotted page architecture** — стандартный паттерн в СУБД.

---

### Как происходит чтение по TID

Когда PostgreSQL получает TID:

1. загружает страницу (page 42)
    
2. идёт в LP[3]
    
3. берёт offset
    
4. читает tuple по этому смещению
    

---

### Вся модель в одной фразе

Можно сформулировать так:

> строка лежит в хвосте страницы,  
> TID указывает на слот (line pointer),  
> слот указывает на реальное положение строки.

---

Если хочешь ещё глубже — можно разобрать:

- бинарный layout tuple (varlena, alignment, null bitmap)
    
- как устроены LP flags (LP_NORMAL, LP_DEAD, LP_REDIRECT)
    
- или как именно VACUUM физически освобождает место и переиспользует LP
