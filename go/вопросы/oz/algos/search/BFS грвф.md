Разберём это так, как ожидают на хорошем собесе — не «что делает BFS», а **почему он даёт кратчайший путь и как именно работает под капотом**.

---

## 🧠 1. Ключевая идея задачи

У тебя граф, где **все рёбра имеют одинаковый вес = 1**.

Это автоматически означает:

> длина пути = количество рёбер

А значит задача сводится к:

> «найти путь с минимальным количеством шагов»

---

## 🔥 2. Почему именно BFS

BFS (поиск в ширину) обходит граф **по уровням (layers)**:

```
start
 ├─ уровень 1 (все соседи)
 │
 ├─ уровень 2 (соседи соседей)
 │
 ├─ уровень 3 ...
```

И это ключевой момент:

> BFS сначала посещает ВСЕ вершины на расстоянии 1,  
> потом ВСЕ на расстоянии 2,  
> потом на расстоянии 3…

👉 Поэтому:

- когда ты впервые дошёл до `target` — это **минимальное возможное расстояние**
    

---

## ⚙️ 3. Что нужно для реализации

Есть 4 базовых структуры:

### 1. Graph (adjacency list)

```go
map[int][]int
```

Чтобы быстро получить соседей вершины.

---

### 2. Queue (очередь)

FIFO структура:

- кладём в конец
    
- достаём из начала
    

Почему именно очередь:

- она гарантирует обработку **по уровням**
    

---

### 3. visited

```go
map[int]bool
```

Чтобы:

- не ходить по кругу
    
- не ломать корректность (иначе можно попасть на более длинный путь)
    

---

### 4. dist / parent

#### dist

Хранит расстояние от start:

```go
dist[child] = dist[parent] + 1
```

#### parent (если нужен путь)

```go
parent[child] = parent_node
```

---

## 🔄 4. Пошаговое выполнение

Допустим граф:

```
1 — 2 — 4 — 5
 \        /
   3 ----
```

Старт: `1`, цель: `5`

---

### Шаг 0

```
queue = [1]
visited = {1}
dist[1] = 0
```

---

### Шаг 1

Берём `1`:

```
соседи: 2, 3
```

Добавляем:

```
queue = [2, 3]
visited = {1,2,3}
dist[2] = 1
dist[3] = 1
```

---

### Шаг 2

Берём `2`:

```
соседи: 1, 4
```

1 уже visited → пропускаем

Добавляем 4:

```
queue = [3, 4]
dist[4] = 2
```

---

### Шаг 3

Берём `3`:

```
соседи: 1, 4
```

оба уже visited → ничего не делаем

```
queue = [4]
```

---

### Шаг 4

Берём `4`:

```
соседи: 2,3,5
```

Добавляем 5:

```
queue = [5]
dist[5] = 3
```

---

### Шаг 5

Берём `5` → это target

👉 стоп

---

## 📏 5. Почему это кратчайший путь (строго)

Потому что:

- BFS обрабатывает вершины в порядке:
    

```
distance = 0 → 1 → 2 → 3 → ...
```

- ты НЕ можешь попасть в вершину с расстоянием 3,  
    не обработав ВСЕ вершины с расстоянием 1 и 2
    

👉 значит более короткого пути просто не существует

---

## 🔁 6. Восстановление пути

Допустим у тебя есть:

```
parent[2] = 1
parent[4] = 2
parent[5] = 4
```

Идёшь назад:

```
5 → 4 → 2 → 1
```

Разворачиваешь:

```
1 → 2 → 4 → 5
```

---

## ⚠️ 7. Важные нюансы

### ❌ Без visited

Будет цикл:

```
1 → 2 → 1 → 2 → ...
```

---

### ❌ Если использовать DFS

DFS может пойти:

```
1 → 3 → 4 → 5 (длинный путь)
```

хотя есть короткий

---

### ❌ Если отметить visited слишком поздно

(после enqueue)

→ можно добавить одну вершину несколько раз  
→ ухудшение до O(E²)

Правильно:

```go
visited[nei] = true
q.PushBack(nei)
```

---

## ⏱️ 8. Сложность

- Время: **O(V + E)**
    
- Память: **O(V)**
    

---

## 🧠 Как рассказать на собесе (идеально)

> «Так как рёбра имеют одинаковый вес, задача сводится к поиску минимального количества шагов. Я использую BFS, потому что он обходит граф по уровням. Это гарантирует, что первое достижение целевой вершины будет кратчайшим путём. Для реализации использую очередь, visited для предотвращения циклов и parent для восстановления пути.»

---

- multi-source BFS
    
- bidirectional BFS (очень любят на собесах)
    
- или переход к Dijkstra и объяснить разницу глубоко
