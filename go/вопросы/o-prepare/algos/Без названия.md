Это задача на external merge sort — когда данные сильно больше памяти. Идея простая: ты не пытаешься сортировать файл целиком, а превращаешь его в набор уже отсортированных кусочков, а потом аккуратно их склеиваешь.

Сначала файл читается последовательно и режется на чанки, которые гарантированно помещаются в RAM. Каждый такой кусок сортируется обычной in-memory сортировкой и записывается во временный файл. В этот момент ты превращаешь одну большую несортированную структуру в множество маленьких уже отсортированных.

Дальше начинается самое интересное — k-way merge. Ты открываешь все эти временные файлы, читаешь по одной строке из каждого и кладёшь их в структуру, которая позволяет быстро находить минимум — обычно это min-heap. Потом в цикле берёшь минимальный элемент, пишешь его в итоговый файл и подгружаешь следующую строку из того же источника обратно в heap. Так ты потоково собираешь финальный отсортированный результат, не загружая всё в память.

Основная оптимизация — это контроль ресурсов. Во-первых, размер чанков: нельзя брать максимум памяти, нужно оставлять запас под строки, буферы и GC, иначе получишь деградацию или OOM. Во-вторых, количество одновременно открытых файлов: если чанков много, ты упрёшься в лимиты дескрипторов, поэтому нужен multi-pass merge — слияние в несколько этапов. В-третьих, I/O: нужно использовать буферизацию и избегать лишних аллокаций, потому что узкое место здесь почти всегда диск, а не CPU.

Есть и менее очевидные моменты. Например, нельзя использовать `bufio.Scanner` из-за ограничения на длину строки — лучше работать через Reader. Важно корректно обрабатывать `\r\n`, иначе можно получить неожиданный порядок. Ещё одна частая проблема — лишние аллокации строк при чтении и записи, что может сильно бить по GC на больших объёмах.

Если говорить про улучшения уровня senior, то добавляют параллельную сортировку чанков, ограничение числа открытых файлов, иногда — повторное использование буферов или даже mmap. Но ключевой архитектурный принцип остаётся тем же: разбить задачу на независимые куски, отсортировать локально и затем эффективно смёржить.

Если коротко формулировать на собесе: это классический external merge sort с контролем памяти, потоковым I/O и k-way merge через heap, с учётом ограничений на файловые дескрипторы и стоимости дисковых операций.

Смысл вопроса — проверить, понимаешь ли ты, что merge делается потоково, а не «загрузили оба куска целиком».

Если у тебя есть 4 GB RAM и ты мёрджишь **два куска по 4 GB**, то ты **не делишь память пополам как 2GB + 2GB и не грузишь их целиком**. Это ошибка мышления.

Правильная модель такая:

ты читаешь **маленькие буферы** из каждого файла и держишь в памяти только их.

То есть:

- из первого файла — небольшой буфер (например, 4–16 MB)
    
- из второго файла — такой же буфер
    
- - буфер для записи результата
        
- - накладные расходы (heap / строки / GC)
        

👉 В итоге каждый файл занимает **мизерную часть RAM**, а не гигабайты.

Если формализовать:

доступная память ≈ 4GB  
тебе нужно:

- buffer₁ + buffer₂ + output_buffer + overhead
    

например:

- 16MB + 16MB + 16MB ≈ 48MB  
    и всё — этого достаточно, чтобы смёржить хоть терабайты.

- при k-way merge память ≈ O(k * buffer_size)
    
- поэтому:
    
    - чем больше k → тем больше RAM
        
    - отсюда появляется multi-pass merge (ограничиваем k)

## 1. Что делает статья (коротко, но по делу)

Алгоритм состоит из двух фаз:

### Phase 1 — Split + Sort (chunking)

- читаем файл **порциями (чанками)**, которые влезают в память (например 100–500 MB)
    
- сортируем chunk в памяти (`sort.Strings`)
    
- пишем его во временный файл
    

👉 на выходе: N отсортированных файлов

---

### Phase 2 — K-way merge

- открываем все временные файлы
    
- читаем по одной строке из каждого
    
- кладём их в **min-heap**
    
- каждый раз:
    
    - берём минимальную строку
        
    - пишем в output
        
    - читаем следующую строку из того же файла
        

👉 это и есть k-way merge

## 5. Heap implementation

```go
type Item struct {
	value string
	idx   int
}

type MinHeap []Item

func (h MinHeap) Len() int { return len(h) }
func (h MinHeap) Less(i, j int) bool {
	return h[i].value < h[j].value
}
func (h MinHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MinHeap) Push(x any) {
	*h = append(*h, x.(Item))
}

func (h *MinHeap) Pop() any {
	old := *h
	n := len(old)
	item := old[n-1]
	*h = old[:n-1]
	return item
}
```

## 6. Merge (стриминг, без загрузки в память)

```go
func mergeFiles(files []string, output string) error {
	out, err := os.Create(output)
	if err != nil {
		return err
	}
	defer out.Close()

	writer := bufio.NewWriter(out)
	defer writer.Flush()

	readers := make([]*bufio.Reader, len(files))
	fds := make([]*os.File, len(files))

	for i, f := range files {
		fd, err := os.Open(f)
		if err != nil {
			return err
		}
		fds[i] = fd
		readers[i] = bufio.NewReader(fd)
	}

	h := &MinHeap{}
	heap.Init(h)

	// initial load
	for i, r := range readers {
		line, err := r.ReadString('\n')
		if err == nil {
			heap.Push(h, Item{value: line, idx: i})
		}
	}

	for h.Len() > 0 {
		item := heap.Pop(h).(Item)

		writer.WriteString(item.value)

		line, err := readers[item.idx].ReadString('\n')
		if err == nil {
			heap.Push(h, Item{value: line, idx: item.idx})
		}
	}

	for _, fd := range fds {
		fd.Close()
	}

	return nil
}
```

## 7. Где это можно улучшить (уровень senior)

### 🔥 1. Параллельная сортировка чанков

```go
worker pool → читаем → сортируем → пишем
```

---

### 🔥 2. Multi-level merge

Если файлов много:

```
1000 chunks →
   merge по 100 → 10 файлов →
   merge → финал
```

---

### 🔥 3. Reuse buffers (важно)

Чтобы не аллоцировать строки:

- использовать `[]byte`
    
- pooling (`sync.Pool`)
    

---

### 🔥 4. I/O batching

- большой буфер writer (например 4–16 MB)
    
- уменьшает syscalls
    

---

### 🔥 5. Backpressure pipeline

Можно сделать pipeline:

```
read → sort → write chunk → merge
```

через каналы
