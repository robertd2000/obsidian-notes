External merge sort — это по сути адаптация merge sort под ситуацию, когда данные не помещаются в RAM и лежат на диске.

Ключевая проблема тут не сама сортировка, а I/O.  
Диск очень медленный по сравнению с памятью, поэтому алгоритм строится вокруг последовательного чтения и минимизации случайных обращений.

Разберём по шагам, что реально происходит.

---

# Фаза 1 — создание отсортированных чанков

Допустим, у тебя есть файл:

```txt
9 1 5 7 2 8 3 6 4
```

Но представим, что памяти хватает только на 3 числа.

Тогда:

## Шаг 1

Читаем первые 3 элемента:

```txt
9 1 5
```

Сортируем в памяти:

```txt
1 5 9
```

Пишем во временный файл:

```txt
chunk1.tmp
1 5 9
```

---

## Шаг 2

Следующие:

```txt
7 2 8
```

↓

```txt
2 7 8
```

↓

```txt
chunk2.tmp
2 7 8
```

---

## Шаг 3

Следующие:

```txt
3 6 4
```

↓

```txt
3 4 6
```

↓

```txt
chunk3.tmp
3 4 6
```

---

Теперь вместо одного огромного несортированного файла у тебя есть:

```txt
chunk1: 1 5 9
chunk2: 2 7 8
chunk3: 3 4 6
```

Каждый chunk уже отсортирован.

---

# Фаза 2 — k-way merge

Вот тут начинается настоящий external merge sort.

Нужно объединить несколько уже отсортированных потоков в один.

---

# Почему нельзя просто читать всё?

Потому что chunks тоже могут быть огромными.

Поэтому читается не весь файл, а только небольшой буфер.

Например:

- по одной строке
    
- или по 4KB
    
- или по 1MB
    

---

# Идея k-way merge

Представим:

```txt
chunk1: 1 5 9
chunk2: 2 7 8
chunk3: 3 4 6
```

Мы читаем только первые элементы:

```txt
1
2
3
```

Теперь вопрос:

> какой из них минимальный?

Очевидно:

```txt
1
```

Пишем его в итоговый файл.

---

# Что дальше?

Важно:

`1` пришёл из `chunk1`.

Значит нужно взять следующий элемент только из `chunk1`.

Теперь в памяти:

```txt
5
2
3
```

Минимум:

```txt
2
```

Пишем.

Берём следующий из `chunk2`.

Теперь:

```txt
5
7
3
```

Минимум:

```txt
3
```

И так далее.

---

# Почему используется min-heap

Если чанков много:

```txt
1000 файлов
```

то каждый раз искать минимум линейно дорого:

```txt
O(k)
```

Поэтому используют min-heap.

Heap позволяет:

- быстро получить минимум
    
- быстро вставить новый элемент
    

Сложность:

```txt
O(log k)
```

---

# Что лежит в heap

Обычно структура типа:

```go
type Item struct {
    Value int
    SourceFile int
}
```

Например:

```txt
heap:
[
  {1, chunk1},
  {2, chunk2},
  {3, chunk3},
]
```

---

# Алгоритм merge

Почти всегда выглядит так:

```txt
1. Открыть все chunk-файлы
2. Прочитать первый элемент каждого
3. Положить их в min-heap
4. Пока heap не пуст:
    - достать минимум
    - записать в output
    - прочитать следующий элемент из того же chunk
    - положить обратно в heap
```

---

# Визуально

Начало:

```txt
chunk1 -> 1 5 9
chunk2 -> 2 7 8
chunk3 -> 3 4 6

heap: [1,2,3]
```

После извлечения `1`:

```txt
output: 1

chunk1 -> 5 9
heap: [2,3,5]
```

После извлечения `2`:

```txt
output: 1 2

chunk2 -> 7 8
heap: [3,5,7]
```

И так далее.

---

# Почему это эффективно

Главное:

## Память почти не растёт

Ты держишь:

- маленький heap
    
- маленькие буферы чтения
    

а не весь файл.

---

# Что реально дорого

Внешняя сортировка почти всегда упирается в:

- disk seek
    
- чтение/запись
    

а не CPU.

Поэтому:

- читают большими блоками
    
- пишут большими блоками
    
- избегают random access
    

---

# Почему merge sort идеально подходит

Потому что merge sort:

1. естественно работает с отсортированными кусками
    
2. требует только последовательного чтения
    
3. не требует случайного доступа
    

QuickSort для диска был бы ужасен.

---

# Что такое "external"

External = данные находятся вне RAM.

Обычно:

- HDD
    
- SSD
    
- distributed storage
    

---

# Реальные применения

External merge sort используется буквально везде:

- PostgreSQL sort spill
    
- Hadoop
    
- Spark
    
- ClickHouse
    
- BigQuery
    
- MapReduce
    
- log processing
    
- merge SSTables в LSM-tree
    

Например ClickHouse активно использует подобные merge-процессы при работе с большими объёмами данных.

---

# Самое важное понимание

External merge sort — это не "одна сортировка".

Это:

## "сортируем маленькие куски в RAM"

↓

## "сливаем отсортированные потоки"

Вся магия именно в потоковом merge без полной загрузки данных в память.
