Составной индекс, в отличие от "одинарного" индекса в качестве индексируемого значения использует не одно значение, а несколько значений то есть если есть 2 поля A и B то составной индекс по ним будет выглядеть примерно так:

```sql
| A | B | 
---------
| 1 | 1 | 
| 1 | 2 | 
| 1 | 4 | 
| 2 | 2 | 
| 2 | 3 | 
| 2 | 5 |
```

Грубо говоря это сортировка по двум полям, а не одному полю. То есть это не индексирование конкатенированных значений A+B.

На уровне BTree это выглядит как будто листочки BTree отвечающие за поле A, ссылаются на вложенное дерево BTree отвечающее за поле B.

Сканирование индекса в вашем случае фактически означает, что поскольку первое поле у вас не задано, то сервер сканирует все листочки дерева A, чтобы найти подходящее значение поля B.

Составной индекс полезен когда информация может быть кластеризована: скажем по этому принципу устроены словари/энциклопедии. А вот если нужна статья заканчивающаяся на "К" - то кластеризация уже не поможет - придется рыться во всей книге (сканировать индекс).

---

**Составной индекс (composite index)** — это индекс, построенный сразу по нескольким колонкам таблицы.

Например:

```sql
CREATE INDEX idx_users_name_age ON users (name, age);
```

---

### 🔹 Как объяснить суть

Он хранит значения **в порядке перечисленных колонок**, то есть фактически индекс отсортирован сначала по `name`, потом по `age`.

---

### 🔹 Ключевое правило (самое важное)

**Порядок колонок в индексе критически важен.**

Индекс `(name, age)` будет использоваться для:

- `WHERE name = 'John'` ✅
    
- `WHERE name = 'John' AND age = 30` ✅
    

Но:

- `WHERE age = 30` ❌ (обычно не используется)
    

👉 потому что работает правило **leftmost prefix**

---

### 🔹 Когда он нужен

Составные индексы используют, когда:

- часто фильтруют по нескольким полям
    
- есть комбинации условий (`WHERE a = ... AND b = ...`)
    
- нужно ускорить сортировку (`ORDER BY`) по нескольким колонкам
    

---

### 🔹 Пример из практики

Таблица заказов:

```sql
(user_id, status, created_at)
```

И частый запрос:

```sql
WHERE user_id = ? AND status = ?
ORDER BY created_at DESC
```

Правильный индекс:

```sql
(user_id, status, created_at)
```

---

### 🔹 Важный нюанс (middle+ уровень)

Составной индекс может покрывать (covering index) запрос, если:

- все нужные поля есть в индексе  
    → тогда не нужно читать саму таблицу
    

---

### 🔻 Как закончить

Составной индекс — это способ оптимизировать запросы по нескольким колонкам,  
но его эффективность полностью зависит от порядка полей и паттернов запросов.

---

## ❓ “Как выбрать порядок колонок в составном индексе?”

Короткий ответ:

Порядок колонок определяется **тем, как выглядят запросы**:  
сначала идут поля с равенством (`=`), потом — с диапазонами (`>`, `<`, `BETWEEN`), затем — под сортировку (`ORDER BY`).

---

Теперь нормальное объяснение, которое ждут:

В PostgreSQL и MySQL составной индекс работает по правилу **leftmost prefix**, поэтому база может эффективно использовать только **левую часть индекса**.

---

### 🔹 Практическое правило

При построении индекса:

1. Сначала колонки из `WHERE` с `=`
    
2. Потом колонки с диапазонами (`>`, `<`, `IN`)
    
3. В конце — колонки из `ORDER BY`
    

---

### 🔹 Пример

Запрос:

```sql
SELECT *
FROM orders
WHERE user_id = 10
  AND status = 'paid'
  AND created_at > now() - interval '7 days'
ORDER BY created_at DESC;
```

Оптимальный индекс:

```sql
(user_id, status, created_at)
```

Почему:

- `user_id`, `status` — точные фильтры
    
- `created_at` — диапазон + сортировка
    

---

### 🔹 Важный нюанс (часто спрашивают)

После диапазонного условия индекс “обрывается”.

Например:

```sql
(a, b, c)
WHERE a = 1 AND b > 10 AND c = 5
```

👉 колонка `c` уже не будет эффективно использоваться

---

### 🔹 Selectivity (сильный уровень)

Иногда порядок выбирают по **селективности**:

- более селективные поля (которые сильнее фильтруют) — раньше
    
- чтобы быстрее сузить выборку
    

Но это вторично по сравнению с паттерном запроса.

---

### 🔹 Частая ошибка

Делать индекс “на глаз”:

```sql
(created_at, user_id)
```

при запросе:

```sql
WHERE user_id = ?
```

👉 такой индекс почти бесполезен

---

### 🔻 Как закончить

Порядок колонок в составном индексе всегда определяется реальными запросами:  
сначала фильтрация, потом диапазоны, потом сортировка — с учётом leftmost prefix.

---

- игнорируют порядок колонок
    
- не связывают индекс с конкретным запросом

## ❓ “Почему индекс не используется, хотя он есть?”

Короткий ответ:

Потому что **оптимизатор считает, что полный скан дешевле**, либо индекс **не подходит под конкретный запрос**.

---

Теперь развернём — это как раз то, где проверяют реальный опыт.

---

### 🔹 1. Не соблюдается leftmost prefix

Есть индекс:

```sql
(a, b)
```

Запрос:

```sql
WHERE b = 10
```

👉 индекс не используется, потому что пропущена первая колонка

---

### 🔹 2. Функции или преобразования

```sql
WHERE LOWER(email) = 'test@mail.com'
```

👉 обычный индекс на `email` не сработает  
(нужен expression index в PostgreSQL)

---

### 🔹 3. Низкая селективность

Если поле почти одинаковое у всех:

```sql
WHERE status = 'active'
```

👉 оптимизатор может выбрать full scan  
потому что индекс не даёт выигрыша

---

### 🔹 4. Маленькая таблица

👉 дешевле прочитать всю таблицу, чем идти через индекс

---

### 🔹 5. Используется диапазон → “обрыв” индекса

Индекс:

```sql
(a, b, c)
```

Запрос:

```sql
WHERE a = 1 AND b > 10 AND c = 5
```

👉 `c` уже не используется

---

### 🔹 6. Неактуальная статистика

Planner принимает решение на основе статистики.

Если она устарела:

- он может выбрать плохой план
    

👉 решается через `ANALYZE` / autovacuum

---

### 🔹 7. Типы не совпадают

```sql
WHERE user_id = '123'
```

если `user_id` — integer

👉 возможен cast → индекс не используется

---

### 🔹 8. LIKE без префикса

```sql
WHERE name LIKE '%abc'
```

👉 B-tree индекс не работает

---

### 🔹 9. OR условия

```sql
WHERE a = 1 OR b = 2
```

👉 индекс может не использоваться (или использоваться частично)

---

### 🔹 10. Покрытие запроса

Если нужно много колонок:

- planner может выбрать seq scan вместо index scan + heap fetch
    

---

## 🔻 Как это диагностировать

Главное, что нужно сказать:

```sql
EXPLAIN ANALYZE
```

👉 смотришь:

- какой план выбран
    
- используется ли Index Scan или Seq Scan
    

---

## 🔻 Как закончить ответ

Индекс не используется либо потому, что он не подходит под запрос (структурно),  
либо потому, что оптимизатор считает его неэффективным на основе статистики.

---

- не знают про EXPLAIN
    
- не понимают роль статистики

## ❓ “Как ты оптимизируешь медленный SQL-запрос?”

Короткий ответ:

Я иду от **измерений → анализа плана выполнения → устранения узких мест**:  
сначала `EXPLAIN ANALYZE`, потом проверяю индексы, переписываю запрос при необходимости и смотрю на данные/статистику.

---

Теперь развернуто — это уже уровень, который ожидают от middle+:

---

### 🔹 1. Сначала понять, что именно медленно

Важно не гадать.

В PostgreSQL:

```sql
EXPLAIN ANALYZE <query>
```

Смотрю:

- где тратится время
    
- какой тип скана (Seq Scan vs Index Scan)
    
- сколько строк читается vs возвращается
    

👉 ключевой момент: **actual vs estimated rows**

---

### 🔹 2. Проверяю индексы

Вопросы, которые я себе задаю:

- есть ли индекс под WHERE?
    
- соблюдён ли порядок в составном индексе?
    
- нет ли full scan из-за отсутствия индекса?
    

Если нужно — добавляю индекс:

```sql
CREATE INDEX ...
```

---

### 🔹 3. Смотрю на сам запрос

Часто проблема не в БД, а в запросе:

- лишние JOIN’ы
    
- `SELECT *` вместо нужных колонок
    
- подзапросы, которые можно переписать
    
- `OR`, который ломает использование индексов
    

Иногда переписываю:

- `JOIN` → `EXISTS`
    
- разбиваю запрос на несколько
    

---

### 🔹 4. Проверяю кардинальность и селективность

Если запрос выбирает 80% таблицы — индекс может не помочь.

👉 тогда:

- либо меняю стратегию
    
- либо оптимизирую структуру данных
    

---

### 🔹 5. Проверяю статистику

Planner зависит от статистики.

Если она устарела:

```sql
ANALYZE;
```

или autovacuum

---

### 🔹 6. Смотрю на сортировки и memory

Если есть:

- `ORDER BY`
    
- `GROUP BY`
    

→ может быть:

- external sort (выход на диск)
    

👉 решается:

- индексом
    
- увеличением work_mem
    

---

### 🔹 7. Проверяю блокировки

Иногда запрос медленный не из-за CPU, а потому что ждёт:

- lock’и
    
- другие транзакции
    

---

### 🔹 8. Думаю про архитектуру

Если запрос тяжёлый по природе:

- денормализация
    
- кэш (например, Redis)
    
- materialized view
    

---

## 🔻 Пример, который хорошо заходит

Запрос медленный → `EXPLAIN ANALYZE` показывает Seq Scan  
→ добавляю индекс → становится Index Scan  
→ время падает в разы

---

## 🔻 Как красиво закончить

Оптимизация — это не “добавить индекс”, а процесс:  
понять план выполнения, найти bottleneck и устранить его на уровне запроса, индексов или архитектуры.

## ❓ “Какие вообще бывают индексы и как уменьшить их размер?”

Короткий ответ:

Помимо обычных B-tree индексов, есть разные типы (GIN, GiST и т.д.), а для оптимизации размера и эффективности используют **partial indexes и expression indexes**.

---

## 🔹 Основные типы индексов (в PostgreSQL)

### 1. B-tree (по умолчанию)

- самый частый
    
- работает для `=`, `<`, `>`, `BETWEEN`, `ORDER BY`
    

---

### 2. Hash

- только для `=`
    
- редко используется (обычно B-tree достаточно)
    

---

### 3. GIN

- для сложных структур
    
- массивы, JSONB, full-text search
    

---

### 4. GiST

- геоданные, диапазоны
    

---

### 5. BRIN

- для очень больших таблиц
    
- хранит “сводку” по блокам
    
- очень маленький по размеру
    

---

## 🔥 То, что ты искал — уменьшение размера

### 🔹 Partial Index (очень важно)

Это индекс **не на всю таблицу, а только на часть строк**.

Пример:

```sql
CREATE INDEX idx_active_users ON users (email)
WHERE deleted_at IS NULL;
```

👉 индекс содержит **только активных пользователей**

---

### Почему это круто:

- индекс меньше → быстрее
    
- меньше I/O
    
- быстрее вставки/обновления
    

---

👉 Это как раз то, что ты вспомнил:  
часто используют условия типа `IS NULL` или `status = 'active'`

---

### 🔹 Expression Index

```sql
CREATE INDEX idx_lower_email ON users (LOWER(email));
```

👉 решает проблему:

```sql
WHERE LOWER(email) = 'test@mail.com'
```

---

### 🔹 Covering Index (INCLUDE)

```sql
CREATE INDEX idx_users_email ON users (email) INCLUDE (name, age);
```

👉 позволяет:

- не ходить в таблицу
    
- ускоряет SELECT
    

---

### 🔹 BRIN (ещё один способ уменьшить размер)

```sql
CREATE INDEX idx_logs_time ON logs USING BRIN (created_at);
```

👉 очень маленький индекс, если данные идут по порядку

---

## 🔻 Как объяснить на собесе

Если тебя спрашивают про “уменьшение индекса”, лучший ответ:

> Я бы использовал partial index — индекс только на нужную часть данных, например `WHERE deleted_at IS NULL`. Это уменьшает размер индекса и ускоряет запросы.

## ❓ “Когда индекс может замедлить систему?”

Короткий ответ:

Индекс ускоряет чтение, но **замедляет запись и потребляет ресурсы**, поэтому при неправильном использовании он может ухудшить производительность.

---

Теперь нормально, как ждут на собесе:

---

### 🔹 1. Замедление INSERT / UPDATE / DELETE

Каждый индекс нужно обновлять при изменении данных.

👉 значит:

- больше индексов → медленнее запись
    
- особенно критично при high-write нагрузке
    

В PostgreSQL это напрямую влияет на:

- latency записи
    
- нагрузку на диск
    

---

### 🔹 2. Избыточные индексы

Если у тебя:

```sql
(a)
(a, b)
(a, b, c)
```

👉 часть индексов дублирует друг друга

- занимают место
    
- замедляют запись
    
- не дают пользы
    

---

### 🔹 3. Индекс не используется

Если индекс не попадает под реальные запросы:

- он просто висит мёртвым грузом
    
- но всё равно обновляется при записи
    

---

### 🔹 4. Низкая селективность

Индекс по полю:

```sql
status = 'active'
```

где 95% строк — active

👉 такой индекс:

- почти не ускоряет SELECT
    
- но замедляет запись
    

---

### 🔹 5. Большой размер индекса

- больше I/O
    
- хуже кеширование
    
- медленнее операции
    

👉 особенно если индекс больше RAM

---

### 🔹 6. Fragmentation / bloat

Индексы тоже раздуваются (как и таблицы):

- хуже производительность
    
- больше чтений
    

👉 лечится через VACUUM / REINDEX

---

### 🔹 7. Неправильный тип индекса

Например:

- B-tree для `LIKE '%abc'` → не работает
    
- индекс есть, но бесполезен
    

---

### 🔹 8. Over-indexing (частая ошибка)

Кандидаты думают:  
“чем больше индексов — тем лучше”

👉 на практике:

- ухудшается write performance
    
- растёт нагрузка на систему
    

---

## 🔻 Как красиво закончить

Индекс — это компромисс:  
мы ускоряем чтение за счёт замедления записи и увеличения использования ресурсов.

Поэтому индекс должен соответствовать конкретным запросам, а не добавляться “на всякий случай”.

---

## ⚠️ Где валятся

- думают, что индекс всегда полезен
    
- не учитывают write overhead
    
- не понимают, что индекс — это тоже данные

## ❓ “Чем отличается clustered index от non-clustered?”

Короткий ответ:

**Clustered index** определяет **физический порядок хранения данных в таблице**,  
а **non-clustered index** — это отдельная структура, которая просто ссылается на данные.

---

Теперь развернуто, как ждут:

---

### 🔹 Clustered index

- данные в таблице **отсортированы по индексу**
    
- сам индекс = сами данные
    

👉 поэтому:

- может быть только **один** clustered index
    
- очень быстрый range scan (`ORDER BY`, `BETWEEN`)
    

---

### 🔹 Non-clustered index

- отдельная структура (обычно B-tree)
    
- хранит:
    
    - ключ
        
    - указатель на строку
        

👉 поэтому:

- может быть много таких индексов
    
- при запросе часто нужно идти:  
    индекс → таблица (heap lookup)
    

---

### 🔹 Пример

Clustered:

```sql
ORDER BY created_at
```

👉 данные уже лежат в нужном порядке

Non-clustered:

```sql
WHERE email = 'test@mail.com'
```

👉 нашли в индексе → пошли в таблицу

---

## 🔥 ВАЖНЫЙ момент (часто ловят)

В PostgreSQL **нет настоящего clustered index** как в других СУБД.

Есть команда:

```sql
CLUSTER
```

но она:

- просто **пересортировывает таблицу один раз**
    
- не поддерживает порядок автоматически
    

👉 PostgreSQL по сути использует heap + индексы

---

### 🔹 Где есть настоящий clustered index

- MySQL (InnoDB)
    
    - primary key = clustered index
        
    - данные хранятся внутри него
        

---

### 🔹 Важное следствие

В MySQL:

- secondary index → хранит ссылку на primary key
    

В PostgreSQL:

- index → хранит pointer на строку (heap tuple)
    

---

## 🔻 Как красиво закончить

Clustered index управляет тем, как данные физически лежат на диске,  
а non-clustered — это просто дополнительный путь для поиска.

И важно учитывать, что PostgreSQL не имеет настоящего clustered index, в отличие от MySQL.

---

## ⚠️ Где валятся

- говорят, что clustered index есть в PostgreSQL (как в MySQL)
    
- не понимают разницу “данные внутри индекса vs отдельно”
    
- не знают, что clustered может быть только один
