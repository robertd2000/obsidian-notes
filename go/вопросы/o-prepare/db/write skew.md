
**Write skew** — это аномалия, которая возникает при snapshot isolation (в PostgreSQL это уровень Repeatable Read), когда две транзакции читают одни и те же данные, принимают решения и записывают изменения, не конфликтуя напрямую, но в итоге нарушают инвариант.

---

Если объяснить на примере, который обычно ждут:

Представь систему дежурств врачей. Есть правило:  
**всегда должен быть хотя бы один дежурный врач**.

В таблице:

```
doctor_id | on_call
1         | true
2         | true
```

Две транзакции начинаются одновременно:

Транзакция A:

- читает: оба врача на дежурстве
    
- решает: “ок, я могу снять врача 1”
    
- делает: `UPDATE doctor SET on_call = false WHERE doctor_id = 1`
    

Транзакция B:

- читает тот же snapshot: оба врача на дежурстве
    
- решает: “ок, я могу снять врача 2”
    
- делает: `UPDATE doctor SET on_call = false WHERE doctor_id = 2`
    

Обе транзакции коммитятся.

👉 итог:

```
1 | false
2 | false
```

Инвариант нарушен — дежурных нет.

---

### Почему это происходит технически

- обе транзакции читают **один и тот же snapshot**
    
- они не видят изменения друг друга
    
- они обновляют **разные строки** → нет прямого конфликта
    
- MVCC не считает это конфликтом
    

👉 поэтому обе проходят

---

### Почему это не ловится в Repeatable Read

Потому что:

- нет dirty read
    
- нет non-repeatable read
    
- но нет контроля **между чтением и записью логики**
    

---

### Как это решается

Есть три подхода:

1. **Serializable уровень**
    
    - PostgreSQL обнаружит конфликт
        
    - одна транзакция упадёт
        
    - нужно делать retry
        
2. **Явные блокировки**
    
    ```sql
    SELECT ... FOR UPDATE
    ```
    
    → блокируем строки, чтобы другая транзакция не могла их менять
    
3. **Constraint / redesign**
    
    - иногда можно выразить инвариант через constraint
        
    - или поменять модель данных
        

---

### Как красиво сформулировать

Write skew — это ситуация, когда транзакции не конфликтуют на уровне строк, но конфликтуют на уровне бизнес-логики, и MVCC этого не видит.

---

## ⚠️ Где валятся

- думают, что Repeatable Read = полная изоляция
    
- не понимают, что конфликт может быть “логический”, а не на уровне строк
    
- не знают про Serializable как решение
