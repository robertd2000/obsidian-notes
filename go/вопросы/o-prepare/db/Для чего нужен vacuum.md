
В PostgreSQL `VACUUM` нужен для очистки “мёртвых” версий строк, которые появляются из-за MVCC.

Когда в PostgreSQL происходит UPDATE или DELETE, старые версии строк не удаляются сразу — они остаются в таблице, потому что могут быть ещё видны другим транзакциям. Со временем таких “dead tuples” накапливается много.

`VACUUM` делает две основные вещи:

- удаляет эти мёртвые версии строк (освобождает место для повторного использования)
    
- обновляет статистику и visibility map, чтобы planner лучше строил планы запросов
    

Важно, что обычный `VACUUM` **не возвращает место операционной системе**, а только переиспользует его внутри таблицы.

Если нужно реально уменьшить размер файла на диске — используется `VACUUM FULL`, но он уже блокирует таблицу.

---

Если чуть глубже (если начнут копать):

Это напрямую связано с MVCC:  
PostgreSQL хранит несколько версий строки, и без `VACUUM` таблицы будут “раздуваться” (bloat), что приводит к:

- замедлению запросов
    
- росту использования диска
    

Поэтому обычно используется **autovacuum**, который автоматически запускается и чистит таблицы без участия разработчика.

---

Короткое резюме одной фразой:

`VACUUM` — это механизм сборки мусора в PostgreSQL, который удаляет неактуальные версии строк и предотвращает раздувание таблиц из-за MVCC.

## 🔹 База (почти всегда)

**“Для чего нужен VACUUM?”**  
— уже разобрали

**“Почему PostgreSQL вообще нуждается в VACUUM?”**  
Ожидают:  
из-за MVCC → старые версии строк не удаляются сразу → нужен garbage collection

---

## 🔹 Чуть глубже

**“Что будет, если не делать VACUUM?”**

Нормальный ответ:

- таблицы начнут раздуваться (bloat)
    
- ухудшится производительность (больше данных читать)
    
- индексы тоже раздуются
    
- в крайнем случае — можно словить wraparound проблему (и это уже критично)
    

---

## 🔹 Autovacuum (очень любят)

**“Что такое autovacuum?”**

→ фоновый процесс в PostgreSQL, который автоматически запускает VACUUM и ANALYZE

---

**“Когда он срабатывает?”**

Ожидают понимание триггера:

- когда количество изменений (UPDATE/DELETE) превышает threshold
    
- threshold = `autovacuum_vacuum_threshold + autovacuum_vacuum_scale_factor * table_size`
    

(формулу знать не обязательно, но упомянуть — плюс)

---

**“Что будет, если отключить autovacuum?”**

Сильный ответ:

- накопятся dead tuples
    
- начнётся bloat
    
- ухудшится performance
    
- и самое опасное — **transaction ID wraparound**, который может привести к остановке базы (Postgres начнёт форсить vacuum)
    

👉 это прям “красный флаг”, если не знаешь

---

## 🔹 Разница VACUUM vs VACUUM FULL

**“Чем отличается VACUUM от VACUUM FULL?”**

Ожидают:

- `VACUUM`
    
    - не блокирует таблицу (или минимально)
        
    - не уменьшает файл на диске
        
    - просто помечает место как свободное
        
- `VACUUM FULL`
    
    - полностью переписывает таблицу
        
    - освобождает место на диске
        
    - **блокирует таблицу (ACCESS EXCLUSIVE lock)**
        

---

## 🔹 ANALYZE

**“Зачем VACUUM ANALYZE?”**

Ответ:

- VACUUM чистит мусор
    
- ANALYZE обновляет статистику
    
- статистика нужна planner’у для оптимальных планов запросов
    

---

## 🔹 Глубже (middle+ / senior)

**“Что такое bloat?”**

→ раздувание таблиц/индексов из-за накопления dead tuples

---

**“Как VACUUM работает под капотом?”**

Достаточно сказать:

- проходит по таблице
    
- проверяет видимость строк (visibility)
    
- удаляет те, которые больше не видны ни одной транзакции
    
- обновляет visibility map
    

---

**“Что такое visibility map?”**

→ структура, которая помогает PostgreSQL понимать, какие страницы:

- не содержат “мёртвых” строк
    
- можно пропускать при VACUUM или использовать для index-only scan
    

---

**“Что такое freeze?” / “зачем freezing?”**

Это уже сильный уровень:

- у каждой строки есть transaction ID
    
- если их не “замораживать” → переполнение (wraparound)
    
- VACUUM делает freeze старых строк, чтобы они считались “всегда видимыми”
    

---

**“Что такое transaction ID wraparound?”**

Коротко:

- XID ограничен (32-bit)
    
- если переполнится → PostgreSQL может начать считать старые данные “новыми”
    
- чтобы этого не случилось → VACUUM обязан делать freeze
    

👉 это один из самых любимых “жёстких” вопросов

---

## 🔹 Практика (очень ценно)

**“Когда ты сам запускал VACUUM вручную?”**

Можно ответить так:

- после массовых DELETE/UPDATE
    
- когда видно сильный bloat
    
- при проблемах с производительностью
    
- перед/после миграций
    

---

**“Как понять, что нужен VACUUM?”**

- рост размера таблицы
    
- падение performance
    
- статистика (`pg_stat_user_tables`)
    
- много dead tuples
    

---

## 🔹 Минимальный набор, который надо знать идеально

Если хочешь “закрыть тему”:

- VACUUM = очистка dead tuples из-за MVCC
    
- autovacuum = автоматизация
    
- VACUUM ≠ VACUUM FULL
    
- есть bloat
    
- есть wraparound (и это критично)
