Отличный список — это прям классический набор вопросов на Go concurrency. Разберу по сути, без лишней теории, но так, чтобы ты мог уверенно отвечать на собесе.

---

# 1️⃣ Примитивы синхронизации в Go

### Основные

**1. `sync.Mutex`**

- эксклюзивный доступ к данным
    

```go
mu.Lock()
x++
mu.Unlock()
```

---

**2. `sync.RWMutex`**

- много читателей / один писатель
    

```go
mu.RLock()
read()
mu.RUnlock()
```

---

**3. `sync.WaitGroup`**

- ожидание завершения горутин
    

```go
wg.Add(1)
go func() {
    defer wg.Done()
}()
wg.Wait()
```

---

**4. `sync.Once`**

- выполнить код один раз
    

```go
once.Do(initFunc)
```

---

**5. `sync.Cond`** (редко)

- сигнализация между горутинами
    

---

**6. `atomic`**

- lock-free операции
    

```go
atomic.AddInt64(&counter, 1)
```

---

**7. Каналы (`chan`)**  
👉 самый идиоматичный способ синхронизации в Go

- передача данных
    
- сигнализация
    
- coordination
    

---

## 💡 Как отвечать на собесе

> В Go есть два подхода: shared memory (mutex/atomic) и message passing (channels). Предпочтительнее channels, если это не ухудшает читаемость.

---

# 2️⃣ Закрытые каналы

```go
close(ch)
```

Используются как:

- сигнал завершения
    
- broadcast (всем слушателям)
    

---

## 🔑 Важное поведение

Чтение:

```go
v, ok := <-ch
```

- `ok = false` → канал закрыт
    

---

## 🧠 Паттерн

```go
for v := range ch {
}
```

→ цикл завершится при закрытии

---

# 3️⃣ Что будет при работе с закрытым каналом

### ❌ Запись

```go
ch <- 1
```

👉 panic: `send on closed channel`

---

### ✅ Чтение

```go
v := <-ch
```

👉 вернёт:

- zero value
    
- не блокируется
    

---

### 💡 Важно

- закрывает канал **только sender**
    
- receiver никогда не должен закрывать
    

---

# 4️⃣ Счётчик из нескольких горутин

## Вариант 1 — atomic

```go
var counter int64
atomic.AddInt64(&counter, 1)
```

✔ быстро  
❌ сложно расширять

---

## Вариант 2 — mutex

```go
mu.Lock()
counter++
mu.Unlock()
```

✔ гибко  
❌ блокировки

---

## Вариант 3 — отдельная goroutine (лучший Go-стиль)

```go
counterCh := make(chan int)

go func() {
    total := 0
    for v := range counterCh {
        total += v
    }
}()
```

👉 это **actor model**

---

## 💬 Как отвечать

> Если нужен простой счётчик — atomic.  
> Если сложная логика — mutex или отдельная goroutine.

# 5️⃣ Context

Контекст — это:

- отмена (`cancel`)
    
- дедлайн (`timeout`)
    
- прокидывание запроса
    

---

## Пример

```go
ctx, cancel := context.WithTimeout(context.Background(), time.Second)
defer cancel()
```

---

## Где использовать

- HTTP запросы
    
- базы данных
    
- goroutines
    

---

## Важно

- всегда передавать первым аргументом
    
- не хранить в struct
    
- всегда вызывать `cancel()`

# 6️⃣ HTTP 301 / 302

Это редиректы.

---

## В Go по умолчанию

```go
http.Get(...)
```

👉 автоматически **следует редиректам**

---

## Если нужно контролировать

```go
client := &http.Client{
    CheckRedirect: func(req *http.Request, via []*http.Request) error {
        return http.ErrUseLastResponse
    },
}
```

---

## Разница

- **301** — постоянный редирект
    
- **302** — временный
    

---

## ⚠️ Подвох

При редиректе:

- могут меняться методы (POST → GET)
    
- важно проверять `Location` header
    

---

# 🔥 Как это связать в один ответ (на собесе)

Если спросят всё сразу:

> В Go есть два основных подхода к синхронизации: через mutex/atomic и через каналы. Каналы позволяют строить пайплайны и избегать shared state. Для координации используется context, который позволяет отменять операции. Закрытые каналы используются как сигнал завершения: чтение из них безопасно, а запись приводит к panic. Для HTTP важно учитывать редиректы — Go по умолчанию их обрабатывает, но это можно переопределить через CheckRedirect.
