## Составные индексы в PostgreSQL: глубокое погружение

Вы уже знаете, что составной индекс — это индекс по двум или более столбцам. Теперь давайте разберём его **детально**: как он устроен внутри, как принимать решения о порядке столбцов, в каких случаях он эффективен, а в каких — бесполезен или даже вреден. Материал построен так, чтобы вы могли уверенно проектировать составные индексы для реальных задач.

---

### 1. Внутреннее устройство: B‑tree для композитных ключей

Стандартный индекс в PostgreSQL — B‑tree. Для составного ключа `(a, b)` в листовых страницах хранятся пары `(a, b) → TID`, **отсортированные** сначала по `a`, а внутри одинаковых `a` — по `b`. Это как телефонный справочник, где записи упорядочены сначала по фамилии, потом по имени.

**Пример содержимого индекса** (упрощённо):
```
(10, 'Иванов')  → TID=...
(10, 'Петров')  → TID=...
(10, 'Сидоров') → TID=...
(20, 'Алексеев') → TID=...
(20, 'Борисов')  → TID=...
...
```

Такая структура позволяет быстро:
- найти все записи с `a = 10` (поскольку они лежат компактным блоком);
- внутри этого блока быстро найти конкретный `b` (бинарный поиск по `b`);
- пройти по диапазону `b` для фиксированного `a`.

**Ключевой момент**: физическая смежность данных по первому столбцу делает индекс чрезвычайно эффективным для запросов, фильтрующих по `a` (одному или в комбинации с `b`), но бесполезным для фильтрации только по `b`.

---

### 2. Правило левого префикса (Leftmost Prefix Rule)

Это фундаментальное ограничение. Индекс `(a, b, c)` может быть использован для условий, которые задействуют **левый префикс** столбцов индекса, идущих подряд.

**Работает для:**
- `WHERE a = 1`
- `WHERE a = 1 AND b = 2`
- `WHERE a = 1 AND b = 2 AND c = 3`
- `WHERE a = 1 AND b > 2` (диапазон по `b` — тоже ок)
- `WHERE a = 1 ORDER BY b` (индекс уже отсортирован по `b` внутри каждого `a`)

**Не работает (или работает очень ограниченно) для:**
- `WHERE b = 2` (первый столбец пропущен)
- `WHERE a = 1 AND c = 3` (пропущен `b` — индекс не может эффективно фильтровать по `c`, так как `c` неупорядочены без `b`)
- `WHERE b = 2 AND c = 3` (пропущен `a`)
- `WHERE c = 3` (пропущены `a` и `b`)

**Почему так?** Без `a` невозможно даже определить, в какой части дерева искать. Без `b` нельзя использовать упорядоченность `c`, потому что после `(a=1, c=...)` могут идти `(a=2, c=...)`, и все значения `c` перемешаны.

---

### 3. Как выбирать порядок столбцов в составном индексе

Это главный практический вопрос. Есть несколько стратегий, которые часто конфликтуют — нужен компромисс.

#### Стратегия 1: «Равенство — потом диапазон»

**Принцип**: столбцы, по которым идёт точное сравнение (`=`), ставьте первыми. Столбцы с диапазонами (`>`, `<`, `BETWEEN`, `LIKE 'foo%'`) — позже.

**Почему?** После того как вы отфильтровали по точным значениям, оставшаяся часть индекса всё ещё упорядочена по следующему столбцу, и диапазон по нему обрабатывается эффективно.

**Пример**:
```sql
CREATE INDEX idx_orders ON orders (customer_id, order_date);
SELECT * FROM orders WHERE customer_id = 123 AND order_date >= '2023-01-01';
```
- Сначала индекс сужается до всех заказов клиента 123 (они лежат компактно).
- Внутри этой группы данные уже отсортированы по `order_date` — можно быстро пройти по диапазону.

Если бы порядок был `(order_date, customer_id)`, то условие `customer_id = 123` не могло бы эффективно использоваться, потому что заказы разных клиентов перемешаны.

#### Стратегия 2: «Наиболее селективный первым»

**Принцип**: Столбец, который сильнее всего фильтрует данные (имеет больше уникальных значений), ставьте первым.

**Пример**: в таблице `users` столбец `country` (200 стран) и `city` (50 000 городов). `city` гораздо селективнее. Индекс `(city, country)` позволит быстрее сузить поиск до небольшого числа строк, чем `(country, city)`.

**Противодействие**: Если вы часто ищете по `country` без `city`, то индекс `(city, country)` не поможет таким запросам. Придётся делать компромисс или два индекса.

#### Стратегия 3: «Учитывайте сортировку (ORDER BY)»

Если запрос часто содержит `ORDER BY b` в дополнение к `WHERE a = ?`, то индекс `(a, b)` даст уже отсортированные данные по `b` внутри каждой группы `a`. Планировщик сможет избежать отдельной сортировки (`Sort`), что очень дорого на больших наборах.

**Пример**:
```sql
CREATE INDEX idx_comments ON comments (post_id, created_at);
SELECT * FROM comments WHERE post_id = 10 ORDER BY created_at;
```
Индекс вернёт записи в правильном порядке, и `ORDER BY` будет выполнен практически бесплатно.

Если же сортировка идёт по убыванию, укажите направление в индексе:
```sql
CREATE INDEX idx_comments_desc ON comments (post_id, created_at DESC);
```

#### Стратегия 4: «Покрытие запроса (INCLUDE)»

Если вы часто выбираете только определённые столбцы, добавьте их в `INCLUDE`. Это не влияет на порядок поиска, но позволяет делать index-only scans.

```sql
CREATE INDEX idx_orders ON orders (customer_id, order_date) INCLUDE (total, status);
```
Теперь запрос `SELECT total, status FROM orders WHERE customer_id = 123 AND order_date = '2023-01-01'` может быть выполнен полностью из индекса, без обращения к таблице.

---

### 4. Когда составной индекс не нужен (или вреден)

- **Запросы фильтруют только по второму столбцу**. Лучше сделать отдельный индекс на `b`.
- **Первый столбец имеет очень низкую селективность** (например, `is_deleted` с 50% `true`). Такой индекс всё равно будет читать много записей. Возможно, лучше индекс на более селективном столбце или частичный индекс.
- **Частые вставки/обновления**. Каждый дополнительный столбец в индексе увеличивает его размер и замедляет DML. Если вы редко используете составной индекс, возможно, лучше обойтись без него или сделать его частичным.
- **Есть уже индекс на левый префикс**. Например, если есть индекс `(a, b)`, то индекс только на `(a)` избыточен, потому что левый префикс уже покрывает запросы по `a`. Удалите лишний.

### 5. Особые возможности и нюансы

#### Смешанные направления сортировки

Если запрос требует `ORDER BY a ASC, b DESC`, обычный индекс `(a, b)` не поможет — сортировка будет не той. Нужно создать индекс с явным указанием:
```sql
CREATE INDEX idx_ab_desc ON table (a ASC, b DESC);
```

#### Индекс на выражении как часть составного

Никто не мешает вам сделать один из столбцов индексом на выражение:
```sql
CREATE INDEX idx_users ON users (lower(email), created_at);
```
Теперь эффективны запросы `WHERE lower(email) = '...'` и комбинации.

#### Сравнение с несколькими индексами

PostgreSQL может использовать **BitmapAnd** / **BitmapOr** для комбинации нескольких отдельных индексов. Например, если есть индексы `(a)` и `(b)`, запрос `WHERE a=1 AND b=2` может использовать оба, затем пересечь их битовые карты. Это часто работает неплохо, но:
- Требует больше ресурсов (чтение двух индексов + битовая операция).
- При большом количестве условий производительность может упасть.
- Не помогает с сортировкой.

Поэтому для пар `(a, b)` с частыми совместными фильтрами составной индекс почти всегда предпочтительнее двух отдельных.

### 6. Диагностика: как понять, используется ли составной индекс

Используйте `EXPLAIN (ANALYZE, BUFFERS)` и обращайте внимание на:
- `Index Scan using idx_name` — хорошо.
- `Bitmap Heap Scan` с `Recheck Cond` — тоже хорошо, но возможно, индекс не очень селективен.
- Если видите `Filter` в плане после `Index Scan` — значит, индексные условия не покрыли всё, и PostgreSQL делает дополнительную фильтрацию. Это сигнал, что порядок столбцов или состав индекса неоптимальны.
- Если видите `Sort` после `Index Scan` — возможно, не хватает столбца для сортировки в индексе.

**Пример плохого плана:**
```
Index Scan using idx_orders on orders
  Index Cond: (customer_id = 123)
  Filter: (order_date >= '2023-01-01')
```
PostgreSQL использовал индекс только по `customer_id`, а `order_date` фильтровал после. Если `order_date` часто в запросах, лучше сделать составной индекс `(customer_id, order_date)` — тогда `Index Cond` будет использовать оба столбца.

### 7. Практические кейсы

#### Кейс 1: Таблица логов событий

Столбцы: `event_type` (10 типов), `created_at` (диапазон), `user_id` (миллионы). Частый запрос: события определённого типа за последний час.

**Индекс:** `(event_type, created_at)`. 
- Первым ставим `event_type` (равенство).
- Вторым `created_at` (диапазон). 
- `user_id` не включаем, так как он редко в фильтрах.

Если бы первым поставили `created_at`, то пришлось бы сканировать огромный диапазон дат, а потом фильтровать по типу — неэффективно.

#### Кейс 2: Таблица заказов с сортировкой

Столбцы: `status` (3 значения), `priority` (1..5), `created_at`. Частый запрос: незавершённые заказы (`status IN ('new','processing')`) с сортировкой по `priority DESC, created_at`.

**Индекс:** 
```sql
CREATE INDEX idx_orders ON orders (status, priority DESC, created_at) 
WHERE status IN ('new','processing');
```
Частичный — исключаем завершённые заказы. Порядок: сначала `status` (равенство), потом `priority` и `created_at` для сортировки. DESC для `priority` важен.

#### Кейс 3: Покрывающий индекс для отчётности

Частый отчёт: `SELECT region, sales, manager FROM deals WHERE date >= '2023-01-01' AND product_id = 42`. Таблица огромная, но строк с product_id=42 немного.

**Индекс:**
```sql
CREATE INDEX idx_deals_covering ON deals (product_id, date) INCLUDE (region, sales, manager);
```
После нахождения нужных строк по `product_id` и `date` все остальные данные уже лежат в индексе. Таблица не читается.

### 8. Резюме: правила для составных индексов

1. **Порядок столбцов критичен.** Сначала те, по которым `=` и `IN`, затем — диапазоны, затем — для сортировки, затем — для покрытия (в `INCLUDE`).
2. **Не пытайтесь включить все столбцы.** Индекс с 5+ столбцами редко эффективен: он большой, медленно обновляется, и шанс, что все они будут использованы в одном запросе, мал.
3. **Используйте `INCLUDE` для покрытия запросов, но не для фильтрации.** Это уменьшит размер индекса по сравнению с включением столбцов в основной состав.
4. **Проверяйте с `EXPLAIN`.** Даже идеально спроектированный индекс может не использоваться из-за неактуальной статистики (запустите `ANALYZE`) или слишком маленькой таблицы (планировщик выберет seq scan).
5. **Не дублируйте префиксы.** Если есть индекс `(a, b, c)`, индекс `(a, b)` обычно избыточен.
6. **Удаляйте неиспользуемые составные индексы** — они только замедляют запись. Мониторьте `pg_stat_user_indexes`.

Составные индексы — один из самых мощных инструментов оптимизации. Правильно спроектированный составной индекс может ускорить запрос в сотни раз, а неправильный — не дать никакого эффекта или даже замедлить систему. Теперь, зная внутреннее устройство и практические стратегии, вы сможете создавать их осознанно.
