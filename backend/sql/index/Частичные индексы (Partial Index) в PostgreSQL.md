## Частичные индексы (Partial Index) в PostgreSQL: глубокое погружение

Вы уже знакомы с общей идеей: частичный индекс — это индекс, который содержит **не все строки таблицы**, а только те, которые удовлетворяют заданному условию `WHERE`. Теперь разберём эту концепцию максимально подробно: как это работает внутри, когда это даёт огромный выигрыш, а когда лучше воздержаться, и какие неочевидные возможности открываются.

---

### 1. Внутреннее устройство: что физически хранится

Обычный индекс на столбце `email` содержит записи для **каждой строки** таблицы. Частичный индекс `CREATE INDEX idx_active_email ON users (email) WHERE active = true` содержит записи **только для тех строк**, где `active = true`. В листьях индекса лежат обычные пары `(значение email, TID)` — но только для активных пользователей.

**Размер**: физически индекс меньше, чем полный индекс на том же столбце, пропорционально доле отфильтрованных строк. Если активных пользователей 5% от всех — индекс будет примерно в 20 раз меньше.

**Обновления**: при вставке новой строки PostgreSQL проверяет условие индекса. Если условие истинно — запись добавляется в индекс. Если ложно — нет. При обновлении `active` с `false` на `true` строка **добавляется** в индекс; при обратном изменении — удаляется из индекса. При обновлении других столбцов (например, `email`) у активной строки запись в индексе обновляется.

**Производительность записи**: частичный индекс быстрее обновлять, чем полный, потому что в нём меньше записей и он занимает меньше места в кэше. Это важно для высоконагруженных систем.

---

### 2. Ключевой концептуальный момент: когда PostgreSQL может использовать частичный индекс

Планировщик использует частичный индекс, только если **условие запроса логически влечёт условие индекса**. То есть: `WHERE` запроса должно гарантировать, что все интересующие нас строки удовлетворяют условию индекса.

**Пример**: индекс `WHERE active = true`. Запрос `SELECT ... WHERE active = true AND email = 'a@b.com'` — очевидно, подходит. Запрос `SELECT ... WHERE email = 'a@b.com'` (без упоминания `active`) — **не подходит**, потому что неизвестно, нужны ли только активные пользователи или все. PostgreSQL не может использовать индекс, так как он не содержит неактивных пользователей, которые могли бы подойти под результат.

**Важное следствие**: частичный индекс эффективен только тогда, когда **ваши запросы постоянно включают в себя его условие**. Если вы иногда ищете без этого условия, понадобится либо полный индекс, либо сканирование таблицы.

### 3. Классические сценарии применения

#### Сценарий 1. Индексирование только «горячего» подмножества данных

Таблица `orders` с миллионами строк. 95% заказов имеют статус `'completed'` или `'cancelled'`. Активные (в работе) — только 5%, но именно по ним чаще всего идут запросы: `SELECT * FROM orders WHERE status IN ('new', 'processing') AND ...`.

Вместо полного индекса на `(status, created_at)` создаём частичный:
```sql
CREATE INDEX idx_active_orders ON orders (created_at) WHERE status IN ('new', 'processing');
```
Индекс в 20 раз меньше. Запросы по активным заказам летают. При этом запросы по завершённым заказам могут использовать другой индекс (или их редкость оправдывает seq scan).

#### Сценарий 2. Уникальность только для части строк

Требование: email должен быть уникальным **только для активных пользователей**. Неактивные (заблокированные, удалённые) могут иметь одинаковые email — это не страшно.
```sql
CREATE UNIQUE INDEX idx_unique_active_email ON users (email) WHERE active = true;
```
Теперь попытка вставить двух активных пользователей с одинаковым email вызовет ошибку. А два неактивных — спокойно.

Это гораздо гибче, чем полная уникальность на всех.

#### Сценарий 3. Исключение «шумовых» значений

В таблице `logs` поле `level`. 99% записей — `INFO` и `DEBUG`. Вас волнуют только `ERROR` и `WARNING`. Индекс на `(timestamp)` с условием `WHERE level IN ('ERROR', 'WARNING')` будет крошечным и позволит молниеносно находить критические события.

#### Сценарий 4. Оптимизация запросов с «редкими» значениями

В таблице `products` столбец `deleted`. По умолчанию большинство товаров активны (`deleted = false`). Запросы на удалённые товары редки. Можно создать индекс на `(category_id, price)` **только для удалённых**:
```sql
CREATE INDEX idx_deleted_products ON products (category_id, price) WHERE deleted = true;
```
Когда нужно найти удалённые товары из какой-то категории — используется маленький индекс. Активные товары индексируются другим способом (или не индексируются по этим колонкам, если это не нужно).

### 4. Ограничения и подводные камни

#### Условие должно быть простым и стабильным

PostgreSQL не позволяет использовать неиммутабельные функции в условии частичного индекса. Например, `WHERE created_at > now()` не сработает — `now()` меняется со временем, и индекс перестанет быть корректным. Нужны условия, которые остаются истинными для фиксированного набора строк на всё время жизни индекса.

#### Не все типы запросов могут использовать частичный индекс

Даже если запрос включает условие индекса, но использует другой план (например, сканирование по другому индексу или bitmap scan), частичный индекс может быть проигнорирован. Нужно проверять через `EXPLAIN`.

#### Комбинация с составными индексами

Можно создавать составные частичные индексы:
```sql
CREATE INDEX idx_active_orders ON orders (customer_id, created_at) WHERE status IN ('new', 'processing');
```
Это мощно: сначала отсекается 95% строк по `status`, затем внутри — индекс по `customer_id` и `created_at`.

#### Сложности с параметризованными запросами (Prepared Statements)

Если вы используете подготовленные операторы с параметрами, например, `WHERE status = $1`, планировщик не знает значения `$1` на этапе планирования. Он может выбрать общий план, который не использует частичный индекс, если для одних значений параметра индекс подходит, а для других — нет. Решение: либо использовать `EXECUTE` с перепланированием, либо отказаться от частичного индекса, если параметры слишком разнообразны.

### 5. Частичный индекс vs. Фильтр в запросе vs. Отдельная таблица

| Подход | Преимущества | Недостатки |
|--------|--------------|-------------|
| **Частичный индекс** | Меньше места, быстрее обновление, прозрачность (одна таблица) | Не работает для запросов без условия индекса |
| **Фильтр в запросе + полный индекс** | Простота, работает для любых запросов | Большой индекс, медленные вставки |
| **Отдельная таблица (партиционирование)** | Полная изоляция, можно разные индексы, очень быстрые запросы по активным | Сложность в поддержке (триггеры/правила), дублирование структуры |

Выбор зависит от частоты использования фильтра и объёма данных.

### 6. Нюансы и продвинутые техники

#### Использование частичного индекса для сортировки

Если условие индекса `WHERE active = true`, а запрос `SELECT ... WHERE active = true ORDER BY created_at`, можно создать индекс `(created_at) WHERE active = true`. Он сразу выдаст отсортированные данные.

#### Частичный индекс на выражении

Комбинация:
```sql
CREATE INDEX idx_upper_email_active ON users (upper(email)) WHERE active = true;
```
Поиск регистронезависимого email только среди активных пользователей.

#### Частичный индекс с INCLUDE

Можно добавить покрывающие столбцы:
```sql
CREATE INDEX idx_active_orders_covering ON orders (created_at) INCLUDE (total, customer_id) WHERE status IN ('new', 'processing');
```

#### «Фильтр-первым» для ускорения сложных запросов

Частичный индекс часто работает как неявный фильтр: сначала индекс отбирает только активные строки, а потом по ним делает остальные операции (join, агрегацию). Это может быть эффективнее, чем фильтровать после join.

### 7. Диагностика: как проверить, используется ли частичный индекс

В `EXPLAIN` вы увидите `Index Cond: (active = true) AND (email = '...')`. Обратите внимание на `Filter` — если условие индекса дублируется как `Filter` после сканирования индекса, возможно, индекс используется не полностью.

Также смотрите на `pg_stat_user_indexes`. Если `idx_scan` для частичного индекса растёт — он полезен. Если долго остаётся нулевым — вероятно, запросы не включают его условие, и индекс можно удалить.

### 8. Практические примеры из реальной жизни

#### Пример A. Таблица пользователей с блокировками

Столбцы: `id`, `email`, `is_blocked`, `blocked_at`. 
- Запросы на вход (логин) ищут только незаблокированных пользователей: `WHERE email = ? AND is_blocked = false`.
- Администраторы иногда ищут заблокированных: `WHERE is_blocked = true AND blocked_at > ?`.

**Индексы:**
```sql
CREATE UNIQUE INDEX idx_login_email ON users (email) WHERE is_blocked = false;
CREATE INDEX idx_blocked_at ON users (blocked_at) WHERE is_blocked = true;
```
Первый индекс маленький и обеспечивает уникальность email только среди активных. Второй — для отчётов по блокировкам.

#### Пример B. Таблица событий с типом "heartbeat"

В системе мониторинга есть события: `heartbeat` (каждую секунду, миллионы записей) и `alert` (редко). Запросы на поиск аварий.

**Индекс:**
```sql
CREATE INDEX idx_alerts ON events (created_at, service_id) WHERE event_type = 'alert';
```
Индекс содержит только аварии. Огромный поток `heartbeat` его не раздувает.

#### Пример C. Мягкое удаление

Таблица `posts` с полем `deleted_at` (NULL для живых). Запросы на главной странице: `WHERE deleted_at IS NULL AND ...`.

**Индекс:**
```sql
CREATE INDEX idx_posts_live ON posts (category_id, published_at) WHERE deleted_at IS NULL;
```
Содержит только живые посты. При удалении поста (простановке `deleted_at`) он исчезает из индекса — что правильно, так как его больше не ищут.

### 9. Когда НЕ стоит использовать частичный индекс

- **Условие индекса истинно для большинства строк** (>80%). Выигрыш в размере будет небольшим, а сложность — дополнительная.
- **Запросы редко включают условие индекса**. Если только 5% запросов используют `active = true`, а остальные — без этого условия, вы всё равно будете либо сканировать таблицу, либо иметь отдельный полный индекс. Частичный индекс не спасёт.
- **Условие индекса сложное или нестабильное**. Например, `WHERE extract(year from created_at) = 2023` — через год индекс устареет морально (строки за 2024 не попадут, но и старые уже не нужны). Лучше использовать партиционирование.
- **Таблица маленькая** (< 10000 строк). Оверхед на проверку условия при каждой вставке может не окупиться, а seq scan и так быстр.

---

### 10. Обслуживание частичных индексов

- **REINDEX** работает как обычно. При `REINDEX CONCURRENTLY` условие сохраняется.
- **VACUUM** учитывает мёртвые строки в частичных индексах так же, как в обычных.
- **ANALYZE** собирает статистику по частичным индексам. Планировщик знает, сколько строк попадает под условие, и может оценивать селективность.

---

### Резюме

**Частичный индекс** — это инструмент для точечного ускорения запросов к подмножеству строк, которое:
- часто используется в фильтрах;
- составляет малую долю от всей таблицы;
- имеет стабильное условие отбора.

**Главная выгода:** уменьшение размера индекса, ускорение DML-операций, возможность наложения уникальности на подмножество.

**Главное ограничение:** запрос должен содержать условие индекса, иначе индекс не будет использован.

**Когда применять:** таблицы с чётко разделёнными «горячими» и «холодными» данными (активные/неактивные, не удалённые/удалённые, ошибки/логи, заказы в работе/завершённые).

Частичные индексы в паре с составными и покрывающими дают гибкость, сравнимую с отдельными материализованными представлениями, но без их накладных расходов на синхронизацию.

Если у вас есть конкретная таблица и запросы — можем вместе спроектировать оптимальный частичный индекс.
