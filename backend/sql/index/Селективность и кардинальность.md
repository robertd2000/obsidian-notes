## Селективность и кардинальность: как база данных оценивает эффективность индексов

Эти два понятия — основа для понимания того, почему один индекс ускоряет запрос, а другой бесполезен. Они тесно связаны и используются планировщиком PostgreSQL для оценки стоимости различных планов выполнения.

---

### 1. Определения

**Кардинальность (cardinality)** столбца — это количество **уникальных значений** в этом столбце.

- Пример: столбец `gender` может содержать значения `'M'` и `'F'`. Его кардинальность = 2.
- Столбец `user_id` в таблице с миллионом пользователей, где каждое значение уникально, имеет кардинальность = 1 000 000.
- Столбец `is_deleted` с двумя возможными значениями (`true`/`false`) имеет кардинальность = 2.

**Селективность (selectivity)** условия — это **доля строк**, которая будет отобрана этим условием. Обычно выражается в виде числа от 0 до 1 (или процентах).

- Высокая селективность = условие отбирает мало строк (близко к 0).
- Низкая селективность = условие отбирает много строк (близко к 1).

Формула для грубой оценки (при равномерном распределении значений):
```
селективность_условия_равенства ≈ 1 / кардинальность_столбца
```

- Для `gender` (кард. 2) → селективность ≈ 1/2 = 0.5 (50% строк).
- Для `user_id` (кард. 1 000 000) → селективность ≈ 1/1e6 = 0.000001 (0.0001%).

---

### 2. Почему это важно для индексов

**Индекс эффективен только для условий с высокой селективностью** (отбирают мало строк). Почему?

- Если индекс отбирает 0.001% строк, он прочитает всего несколько записей в индексе и сделает пару обращений к таблице — быстро.
- Если индекс отбирает 50% строк, он прочитает половину всех записей индекса (что уже много) и для каждой сделает случайное обращение к таблице. В этом случае гораздо дешевле прочитать всю таблицу последовательно (seq scan), пропуская постраничные операции ввода-вывода.

**Пример из жизни:**  
Таблица `users` (10 млн записей). Запрос `SELECT * FROM users WHERE gender = 'M'`.  
- Селективность ~50% (около 5 млн строк).  
- Индекс по `gender` не поможет — планировщик выберет seq scan, потому что чтение 5 млн строк через индекс (с 5 млн случайных обращений к таблице) будет чудовищно медленным.

Запрос `SELECT * FROM users WHERE user_id = 123456`:
- Селективность ~0.0001% (1 строка).
- Индекс по `user_id` идеален: одно чтение индекса + одно чтение таблицы.

### 3. Как PostgreSQL узнаёт кардинальность и селективность

Планировщик не гадает — он полагается на **статистику**, собранную командой `ANALYZE` (или автоанализом). Статистика хранится в системном каталоге `pg_statistic` (доступна через представление `pg_stats`).

Для каждого столбца `ANALYZE` собирает:
- **`n_distinct`**: оценка кардинальности (если отрицательное значение — это доля уникальных значений, например -0.1 означает, что уникальных значений примерно 10% от всех строк).
- **`most_common_vals` / `most_common_freqs`**: список самых частых значений и их частоты (например, пол `'M'` встречается в 49.5% строк).
- **`histogram_bounds`**: границы гистограммы для распределения значений (для числовых и дат).
- **`null_frac`**: доля NULL-значений.

На основе этих данных планировщик оценивает селективность любого условия.

Пример запроса к статистике:
```sql
SELECT attname, n_distinct, most_common_vals, most_common_freqs 
FROM pg_stats 
WHERE tablename = 'users' AND attname = 'gender';
```

### 4. Нюансы: неравномерное распределение и корреляция

Реальная селективность может сильно отличаться от `1 / кардинальность`, если распределение неравномерно.

**Пример:** столбец `country`. Кардинальность = 200 (200 стран), но 90% пользователей из США.  
- Селективность `country = 'USA'` ≈ 0.9 (очень низкая селективность — 90% строк). Индекс бесполезен.
- Селективность `country = 'Vanuatu'` ≈ 0.00001 (очень высокая). Индекс отлично подойдёт.

PostgreSQL знает об этом из `most_common_freqs`. Для `'USA'` частота будет 0.9, и планировщик не станет использовать индекс.

**Корреляция (correlation)** — ещё один важный параметр из `pg_stats`. Он показывает, насколько физический порядок строк в таблице соответствует логическому порядку значений столбца.  
- Корреляция ≈ 1 — значения на диске идут в том же порядке, что и в индексе. Тогда index scan может быть очень эффективным (близким к последовательному чтению).  
- Корреляция ≈ 0 — значения перемешаны. Index scan вызовет много случайных обращений и будет медленным. Для диапазонных запросов на таких столбцах может оказаться выгоднее seq scan, даже если селективность не очень высока.

### 5. Практические правила для разработчика

1. **Создавайте индексы на столбцах с высокой ожидаемой селективностью** (мало уникальных значений — плохо, много — хорошо).
2. **Проверяйте реальную селективность** через `EXPLAIN ANALYZE`. Если планировщик ошибается (например, думает, что строк 1, а их 1000), обновите статистику (`ANALYZE`) или настройте параметры сбора статистики (увеличьте `default_statistics_target`).
3. **Для низкоселективных условий используйте другие методы**: частичные индексы, фильтрацию только по индексу (covering), или пересмотрите логику запроса.
4. **Помните про NULL**: для условий `IS NULL` селективность может быть особенной. Если NULL-значений мало, индекс может быть эффективен (особенно если создать частичный индекс `WHERE col IS NULL`).
5. **Составные индексы**: порядок столбцов должен отражать селективность — сначала самые селективные (при прочих равных). Но помните про правило левого префикса — часто приходится искать компромисс.

---

### 6. Типичные ошибки

- **Индекс на столбце типа `boolean`** (кардинальность 2). Такой индекс почти никогда не используется, разве что в сочетании с другими столбцами или как частичный.
- **Индекс на `status` с 3-4 значениями**, если распределение равномерное. Бесполезен, если вы ищете по всем значениям. Но может быть полезен для поиска редкого статуса (например, `status = 'error'`), если он встречается в 0.1% случаев.
- **Игнорирование статистики**: после массового изменения данных (большие вставки, удаления) статистика устаревает, планировщик строит неверные оценки. Регулярный `ANALYZE` (или autovacuum) обязателен.

---

### Резюме

- **Кардинальность** — сколько уникальных значений.  
- **Селективность** — какая доля строк будет отобрана.  
- Индекс эффективен при **высокой селективности** (мало строк).  
- Планировщик опирается на статистику, собранную `ANALYZE`.  
- Низкая кардинальность ≠ автоматически плохо, если распределение сильно неравномерно (редкие значения можно индексировать).  
- Проверяйте `pg_stats` и `EXPLAIN`, чтобы понимать, что происходит на самом деле.

Это база, на которой строится любая серьёзная оптимизация индексов в PostgreSQL. Если хотите, можем разобрать конкретный пример с реальными цифрами и объяснением плана запроса.
