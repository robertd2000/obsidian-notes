Данные в PostgreSQL хранятся на диске в виде файлов, разбитых на страницы фиксированного размера. Разобраться в этой системе — значит понять ключевые концепции, такие как куча, многоверсионность, индексы и вспомогательные карты. Если объяснять очень кратко, то данные в PostgreSQL хранятся в виде «кучи» (heap) строк в таблицах, разделенных на страницы, а для быстрого доступа к ним используются индексы, которые по сути являются отдельными структурами, указывающими на физическое расположение строк в этой куче.

Давайте теперь разберем всё по порядку на основных уровнях.

### 🗂️ Физическая организация: от каталога до страницы

На самом высоком уровне вся база данных хранится в директории, называемой `PGDATA`. Внутри нее данные логически разделены:

*   **`base/`**: Основная директория, где хранятся файлы всех пользовательских баз данных. У каждой базы данных есть своя поддиректория, идентифицируемая её внутренним числовым идентификатором (OID).
*   **`global/`**: Содержит общие для всего кластера базы данных системные каталоги, например, список пользователей или информацию о табличных пространствах.
*   **`pg_wal/`**: Директория для журнала предзаписи (Write-Ahead Log, WAL), который обеспечивает надежность и восстановление данных после сбоя.

Каждая таблица или индекс в базе данных — это один или несколько файлов в файловой системе. Чтобы обойти ограничения файловых систем, эти файлы разбиваются на сегменты размером 1 ГБ. Как только таблица или индекс превышают этот размер, для них создается новый сегмент с суффиксом `.1`, `.2` и так далее.

Самый нижний и фундаментальный уровень организации данных — это **страница** (page), которая обычно имеет размер **8 КБ**. Это атомарная единица, с которой работает PostgreSQL при чтении и записи на диск. Каждая таблица и каждый индекс представляют собой массив таких страниц.

### 📄 Внутреннее устройство страницы данных (на примере кучи)

Давайте заглянем внутрь страницы таблицы (кучи). Её структура тщательно продумана для эффективности. Вот основные компоненты:

*   **Заголовок страницы (PageHeaderData)**: Занимает 24 байта в начале страницы и содержит важнейшую служебную информацию:
    *   `pd_lsn`: Указывает на последнюю запись в WAL, которая изменила эту страницу, что критически важно для восстановления.
    *   `pd_checksum`: Контрольная сумма для проверки целостности страницы, если эта функция включена.
    *   `pd_lower` и `pd_upper`: Указатели на начало и конец свободного пространства.
    *   `pd_special`: Указатель на специальную область, которая используется для хранения специфичных для индексов данных (в таблицах она не используется).
*   **Массив указателей на строки (ItemIdData)**: Это массив из 4-байтовых элементов, каждый из которых является указателем (содержит смещение и длину) на физическое расположение строки (кортежа) на странице. Этот массив растет от начала страницы.
*   **Область данных (Items)**: Здесь физически, строка за строкой, хранятся сами строки (кортежи). Они добавляются с *конца* страницы и растут в направлении начала.
*   **Свободное пространство (Free space)**: Это область между массивом указателей и областью данных. Именно сюда будут записаны новые строки или обновленные версии существующих.

Страница кучи может выглядеть так:
`[ЗАГОЛОВОК][УКАЗАТЕЛЬ1][УКАЗАТЕЛЬ2]... -> свободное место <- [ДАННЫЕ2][ДАННЫЕ1]`

Физический адрес любой строки в таблице — это комбинация номера страницы и индекса (смещения) в массиве указателей. Этот адрес хранится в системном столбце `ctid` и имеет формат `(страница, индекс)`.

### 🍲 Куча (Heap) и MVCC: Как живут строки

Таблица в PostgreSQL — это неупорядоченный набор строк, называемый **кучей (heap)**. Ключевая особенность PostgreSQL — это механизм многоверсионности (MVCC). Когда вы обновляете (`UPDATE`) или удаляете (`DELETE`) строку, PostgreSQL не изменяет её на месте. Вместо этого:

1.  Создается **новая версия** строки.
2.  Старая версия помечается как «мёртвая» (dead tuple), но физически остается на странице.
3.  Индексы начинают указывать на новую версию.
4.  Через некоторое время специальный фоновый процесс **autovacuum** находит мёртвые строки и помечает занимаемое ими место как свободное для переиспользования.

Управление версиями осуществляется с помощью системных столбцов, которые PostgreSQL добавляет к каждой строке:
*   **`xmin`** и **`xmax`**: Идентификаторы транзакций, которые создали и удалили (или обновили) данную версию строки.
*   **`ctid`**: Физический адрес версии строки.
*   И другие, например, для TOAST.

### 🗺️ Вспомогательные карты: FSM и VM

Для эффективного управления страницами PostgreSQL использует два вспомогательных файла для каждой таблицы:

*   **Карта свободного пространства (Free Space Map, `.fsm`)**: Это небольшая структура данных, которая отслеживает, сколько свободного места осталось на каждой странице кучи. Когда нужно вставить новую строку, PostgreSQL обращается к FSM, чтобы быстро найти страницу с достаточным количеством места, избегая сканирования всех страниц таблицы.
*   **Карта видимости (Visibility Map, `.vm`)**: Эта карта хранит по два бита на страницу таблицы. Её главная задача — отмечать страницы, на которых все строки являются видимыми для всех текущих и будущих транзакций. Это дает два больших преимущества:
    1.  **Ускорение VACUUM**: `VACUUM` может пропускать полностью видимые страницы, так как на них заведомо нет «мёртвых» строк.
    2.  **Ускорение сканирований только по индексу (Index-Only Scans)**: Если все строки на странице видны, PostgreSQL может безопасно вернуть данные из индекса, не обращаясь к самой таблице для проверки видимости, что значительно ускоряет запросы.

### 🗂️ Индексы (на примере B-Tree)

Индекс — это **отдельная** структура данных, физически не зависящая от таблицы, но содержащая ссылки на неё. Наиболее распространенный тип индекса в PostgreSQL — B-Tree. Вот как он устроен:

*   **Структура**: Это многоуровневое сбалансированное дерево, где каждый узел — это страница (обычно 8 КБ).
*   **Метастраница (Metapage)**: Это специальная первая страница индекса, которая хранит служебную информацию о корне дерева и другие метаданные.
*   **Корневые и внутренние страницы (Root & Internal pages)**: Содержат «направляющие» кортежи, которые указывают на более низкие уровни дерева (на другие страницы).
*   **Листовые страницы (Leaf pages)**: Это самый нижний уровень B-Tree, где и находится основная информация. В отличие от внутренних узлов, листовые страницы содержат не ключи-указатели, а сами пары «**значение индекса → TID**» — то есть значение, по которому мы ищем, и физический адрес (`ctid`) соответствующей строки в таблице. Кроме того, листовые страницы связаны между собой в двусвязный список, что позволяет эффективно сканировать диапазоны значений.

Как B-дерево обрабатывает изменения:
1.  При вставке нового ключа PostgreSQL находит нужную листовую страницу.
2.  Если на ней есть место, ключ и `TID` просто добавляются.
3.  Если страница заполнена, она **разделяется (page split)**. Часть данных переносится на новую страницу, и в родительскую страницу добавляется новый указатель. Этот процесс может «каскадироваться» вплоть до корня, и, если потребуется, даже создать новый корневой уровень.

### 💎 TOAST: Работа с большими полями

PostgreSQL имеет ограничение на размер строки — примерно 8 КБ (размер страницы). Для полей, которые могут быть очень большими (`TEXT`, `VARCHAR`, `BYTEA`, `JSONB`), используется механизм **TOAST (The Oversized-Attribute Storage Technique)**.

Когда строка превышает примерно **2 КБ**, большие поля сжимаются (обычно с помощью LZ-сжатия) и/или выносятся в специальную связанную таблицу `pg_toast`. В основной таблице остается лишь указатель на эти данные. Благодаря этому основная таблица остается компактной и эффективной для сканирования.

### ⚙️ Настройка: Fillfactor

Параметр `fillfactor` для таблицы определяет, насколько плотно будут заполняться страницы при вставке новых данных. По умолчанию он равен `100` (100%). Уменьшение значения, например, до `80`, оставляет 20% страницы свободными. Это пространство специально резервируется для оптимизации **HOT (Heap-Only Tuples)**.

HOT-обновления — это особый тип обновления, когда изменяются **неиндексированные** столбцы. В этом случае новая версия строки может быть помещена на ту же страницу, что и старая, и указатель на нее прописывается прямо в старой версии. Это позволяет **полностью избежать создания новой записи в индексе**, что значительно сокращает разрастание индексов и нагрузку на `VACUUM`. Резерв свободного места, задаваемый через `fillfactor`, критически важен для эффективности HOT-обновлений.
